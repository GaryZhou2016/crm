Summary

September 29, 2019 by Gary

This dummy project demonstrates a few areas of development in frontend and backend. As it is limited by scope, I have shown the basic more than a production completeness. Therefore, I write this summary to summarize what have done and make comments for possible improvements. Development details are also included.

Briefly, this project implemented microservices to allow CRUD operations to the data store. While frontend consume the services and present data through web pages. It is important to decouple data, business logic and presentations.

Backend

What backend provides is the WESTFUL API for data munipulations.

  1. Database: MySQL. Schema is created with CRM_Table.sql. There are three tables: Customer, Office, and their relation table. A customer can have multiple offices and a office can have many customers. After installing MySQL, there are three parameters url, username, password need to be set. Populating test data into the three tables can be done easily through sql editor.
  2. The OM tools are myBatis and Spring.
  3. Three APIs are implemented for DRUD operations.
  4. Tomcat I use 8.5 and set the port to 8181.
  5. JDK I use 10 and Eclipse 2019.

Comments

  • Although foreign key constraints are set, cascade delete is still used for simplicity.
  • Backend storage is a simple setup for this project. No clustering is involved. In fact, the CRUD service here implemented could simply be a reverse proxy in some HA and distributed environments.
  • The CRUD service can be implemented by Node.js. This project use Java technology stack showing microservice can be independent of languages.

Frontend

Frontend is a simple web app in Angular. The implementation carefully maintains separations.

  1. There are 3 pages including this one as a front page. They are Introduction, Customers, Offices.
  2. There are two services have been implemented to consume the APIs.
  3. On page Customers, the GET operation on the table is shown. A customer can have 0 to n offices. This page shows the to-n relationship.
  4. On page Offices, all the 4 CRUD operations are applied. The page is self-explained.

Comments

  • The APIs uri are hard coded. It could be configurable or through a service discovery agent. For example, a DNS. Also they are kept inside the services. Down stream consumers only ask for service and having no information about it.
  • There could be more tests implemented.
